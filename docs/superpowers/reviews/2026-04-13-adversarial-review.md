## Executive summary

**Verdict across all six phases: needs-attention.** Every phase returned at least one high-severity finding. Codex found **16 distinct findings total** (1 critical, 13 high, 2 medium), with clear clustering around four structural problems rather than scattered defects.

**The four structural problems, ranked by blast radius:**

### 1. tapd's event / notification pipeline is half-wired (F1.3 + F4.2 + F5.2)

This is the biggest issue and it cuts across Phases 1, 4, and 5. The event bus works, but:
- `TapdRuntime.translate()` invents fields that `TapMessagingService.emitEvent` never emits (`actionKind`, `conversationId`, `txHash` on inbound results, `connectionId` on connection results). Any SSE consumer gets mis-shaped events for scheduling requests, action results, and connection results.
- `/api/notifications/drain` returns from a `NotificationQueue` that nothing ever enqueues. The Hermes Python plugin and the OpenClaw `before_prompt_build` hook both call this endpoint and always get an empty list in production.
- The OpenClaw escalation watcher only listens for `action.pending` and `connection.requested`, but `TapdRuntime` never emits those types. Even if the queue worked, the watcher would never fire.

**Impact:** pending approvals, inbound messages, and scheduling escalations are invisible to any host that relies on the tapd drain hook (Hermes and OpenClaw both do). This is the "plumbing is there, pipes aren't connected" class of problem.

**Fix shape:** redesign event translation at the source. Either enrich `TapMessagingService.emitEvent` with structured metadata, or translate from actual emitted fields (scope, parsed protocol payloads, journal metadata). Wire a producer from the event bus into `NotificationQueue` with classification rules. Emit the event types the OpenClaw watcher expects. Cover the end-to-end path (runtime event → bus → queue → drain) with an integration test against a real in-process tapd.

### 2. Route/client contract mismatches between tapd and host plugins (F4.3 + F5.3 + F5.4)

Multiple host clients send bodies tapd doesn't accept, or drop fields tapd should accept but doesn't:
- **`request_meeting`** — both the Hermes Python client and the OpenClaw TS client send a flat `{ peer, title, duration, preferred, location }` shape; tapd's `/api/meetings` wants a full `SchedulingProposal` with `type`, `schedulingId`, `slots`. Every Hermes and OpenClaw meeting request 400s.
- **`request_funds` / `transfer`** — the Hermes client omits defaults for `chain` and `toAddress` that the old flow supplied, so commands that used to work now fail.
- **`autoGenerated`** — the OpenClaw tool advertises it, the client forwards it, tapd's `/api/messages` silently drops it. Auto-reply protection is broken.

**Impact:** three user-facing features (scheduling, default-chain transfers, auto-reply loop prevention) are regressed.

**Fix shape:** add route-compatibility tests for every `tap_gateway` action against the real tapd handler (not a fake stub). Make the client either send the same shape as the CLI did, or extend tapd's routes to accept the flat shape and build the proposal internally. Extend `/api/messages` to accept `autoGenerated`.

### 3. Trust / convergence regressions (F3.1 + F2.3 + F5.1)

The refactor introduced three separate trust-boundary issues:
- **`tap contacts remove` offline** bypasses the `connection/revoke` wire message and deletes local state unilaterally. The spec's own "recovery invariant" is revoke-before-delete; the new fallback breaks it.
- **UI pending-thread matching** keys on `peerAgentId` alone, not `(peerAgentId, peerChain)` or `connectionId`. Cross-chain collisions can route an approval to the wrong thread, and the operator can approve the wrong action.
- **OpenClaw socket discovery** ignores `TAP_DATA_DIR` entirely. An operator running tapd under an isolated data dir will silently talk to their default daemon instead of the intended one. Cross-agent isolation is broken.

**Impact:** trust state can silently diverge between peers; approvals can hit the wrong counterparty; multi-agent isolation doesn't hold.

**Fix shape:** fail-closed on offline revoke. Use `connectionId` as the canonical join key in the UI. Standardize socket discovery across all host plugins with the exact same precedence order (`socketPath > dataDir > TAP_DATA_DIR > ~/.trustedagents`), and test each one.

### 4. Startup / lifecycle / migration hazards (F1.1 + F3.2 + F3.3 + Fv2.1 + Fv2.2)

Several places assume the happy path and strand state on failure:
- **`Daemon.start()`** starts the transport before binding HTTP. HTTP bind failure leaves the transport owner lock held with no control plane.
- **`tap daemon start|stop|restart`** never verifies the right process actually started or stopped — stale port files and abandoned PIDs cause split-brain.
- **`packages/tapd/src/bin.ts`** binds the `OwsSigningProvider` to the startup `trustedAgentsConfig.chain`, so cross-chain transfers (Taiko, the only other documented chain) regress.
- **SQLite migration** marks itself complete even when individual files fail to import, and rebuilds non-empty conversations through the logMessage path which loses `startedAt` / `status` / original chronological ordering.

**Impact:** a single bad startup, stale file, or malformed legacy JSON can permanently strand state, silently break a chain, or leave a conversation mis-ordered forever.

**Fix shape:** make `Daemon.start()` transactional (unwind on failure). Verify the spawned PID is bound before reporting start success; verify the pidfile process is gone before reporting stop success. Load chain config per-request or at least honor the full CLI chain map. Make the migration fail-closed: archive/flag only when every file imported cleanly.

## Phase 1 — tapd daemon

**Target:** `f108921..1764f19`
**Verdict:** needs-attention

No-ship. This branch introduces a tapd startup failure mode that can strand the transport lock, regresses Hermes scheduling escalations, and publishes corrupted action metadata on the new SSE surface.

### Findings

**[high] F1.1 — Failed tapd startup can leave the XMTP runtime alive with no reachable API**
`packages/tapd/src/daemon.ts:52-83`

`Daemon.start()` starts `TapMessagingService` before binding the HTTP listeners. If `TapdHttpServer.start()` then throws (port already in use, stale socket path, permission error), the method exits without stopping the just-started runtime. `TapMessagingService.start()` acquires the transport owner lock and may already have the transport running, so a failed daemon boot can still monopolize the agent data dir while exposing no control plane. The likely operator experience is "tapd failed to start" followed by every retry or fallback host seeing the lock and refusing to run until the process is killed.

**Recommendation:** Make startup transactional: hold the service/server in locals, and on any failure after `runtime.start()` call `runtime.stop()` (and `server.stop()` if needed) before rethrowing. Add a regression test that forces TCP or socket bind failure and asserts the owner lock is released.

**[high] F1.2 — Hermes no longer surfaces queued scheduling requests to the operator**
`packages/cli/src/hermes/event-classifier.ts:1-5`

This re-export replaces Hermes' classifier with the core one, but Hermes still has no explicit scheduling notification hook. In `packages/cli/src/hermes/registry.ts:568-572`, `approveScheduling` just returns `null`, and `handleEmitEvent()` only notifies when `classifyTapEvent()` returns a bucket (`registry.ts:710-722`). The new classifier returns `null` for queued `action/request` events, so inbound meeting proposals now disappear from `[TAP Notifications]` and can sit pending with no operator prompt. The deleted `packages/cli/test/hermes-event-classifier.test.ts` was covering exactly this behavior.

**Recommendation:** Restore Hermes-specific handling for queued scheduling requests, or mirror OpenClaw by enqueueing a notification inside `approveScheduling`. Keep a regression test for queued scheduling events so this path cannot silently regress again.

**Note:** Phases 4 deletes `packages/cli/src/hermes/` entirely, so this specific file is moot — but the underlying behavior gap (Hermes scheduling escalations disappearing) may have been carried forward into the tapd notification queue if the classifier is still the only decision point. Confirm in Phase 4 review.

**[high] F1.3 — tapd's event bridge fabricates action metadata the core runtime never emits**
`packages/tapd/src/runtime.ts:102-141`

`TapdRuntime.translate()` assumes raw `emitEvent` payloads contain `actionKind`, `conversationId`, `connectionId`, `txHash`, and that `payload.id` is the request being completed. The core service does not emit that shape on the normal action paths: scheduling requests are exposed via `scope: "scheduling/request"` only (`packages/core/src/runtime/service.ts:2474-2477`), and inbound `action/result` events are wrapped by `emitIncomingAndReturn()` with only method/id/fromName (`service.ts:2568-2570`). In this translator, queued scheduling requests are therefore published as `kind: "transfer"`, results lose their original `requestId`/conversation context, and `connection/result` events usually get an empty `connectionId`. Any SSE/UI consumer will mis-render action cards and cannot reliably correlate a completion back to the pending request it is supposed to update.

**Recommendation:** Stop defaulting missing fields into a typed event. Either enrich `TapMessagingService.emitEvent` at the source with structured action metadata, or translate from actual emitted fields (`scope`, parsed protocol payloads, request-journal metadata) before publishing to the bus. Add SSE tests for scheduling requests/results and connection results, not just plain messages.

### Next steps
- Make `Daemon.start()` unwind partial startup and add a bind-failure regression test.
- Restore Hermes scheduling escalation delivery and reintroduce the deleted classifier coverage (may already be fixed by Phase 4 removing hermes/, confirm during Phase 4 review).
- Redesign tapd event translation around fields the core service actually emits, then add SSE tests for scheduling and result correlation.

## Phase 2 — web UI

**Target:** `1764f19..46bd2f9`
**Verdict:** needs-attention

No-ship: the pending-action path breaks on real `/api/pending` data, stale auth bricks the dashboard after daemon restarts, and pending approvals can be shown under the wrong peer when agent IDs collide across chains.

### Findings

**[high] F2.1 — Real pending requests crash the approval UI**
`packages/ui/components/chat/PendingActionCards.tsx:31-36`

`/api/pending` does not return an action kind here; core/tapd expose `kind: "request"` for pending journal entries, and this component blindly casts that string to `ActionKind`. `ActionCard` then dereferences `KIND_META[kind]`, so the first real pending item renders `meta` as `undefined` and the thread blows up instead of showing approve/deny controls. This is user-visible on the exact flow the change is trying to add.

**Recommendation:** Stop using `item.kind` for UI card selection. Derive the card type from `item.details?.type`, or add a dedicated backend field like `actionKind`, and guard unknown values before rendering `ActionCard`.

**[high] F2.2 — Daemon restarts leave the dashboard stuck on an infinite loading screen**
`packages/ui/components/Dashboard.tsx:43-48`

The dashboard reads only SWR `data` and never handles fetch errors; later in the same component, missing `identity` is treated as "Loading…" forever. That is unsafe here because tapd generates a fresh auth token on every `Daemon.start()` while the UI persists the previous token in `sessionStorage`. After any daemon restart or a reload with a stale token, every request becomes 401, `identity` never resolves, and the operator gets a permanent spinner with no recovery path.

**Recommendation:** Handle 401s centrally in the API/SWR layer: clear the stored token, tear down SSE, and bounce back to an explicit re-auth/no-token state instead of treating auth failure as a loading condition.

**[medium] F2.3 — Pending approvals are associated to a thread by agent ID alone**
`packages/ui/components/Dashboard.tsx:116-121`

This filter binds pending requests to the selected DM using only `peerAgentId`. Core explicitly treats `(peerAgentId, peerChain)` as the stable peer identity because numeric token IDs can collide across chains, and pending request details already carry `peerChain`. In a multi-chain setup, a Base request and a Taiko request for the same agent ID can land in the same thread, so the operator can approve the wrong action from the wrong conversation.

**Recommendation:** Plumb `connectionId` through `TapPendingRequest` and filter by that; if that is not available, at least scope pending items by both `peerAgentId` and `details.peerChain` against the selected contact.

### Next steps
- Plumb a canonical action kind into pending responses and add a UI test that renders a non-empty `/api/pending` payload.
- Add explicit stale-token handling for 401s so a daemon restart returns the user to a recoverable state instead of a spinner.
- Scope pending-thread matching by full peer identity (`connectionId` or `peerAgentId` + `peerChain`) and cover it with a multi-chain regression test.

## Phase 3 — CLI thin-client

**Target:** `46bd2f9..80b6d2c`
**Verdict:** needs-attention

No-ship: this refactor breaks a core convergence guarantee in `tap contacts remove`, makes the new daemon control commands report success without proving the right process actually started/stopped, and freezes tapd onto the startup chain/config so cross-chain thin-client flows regress.

### Findings

**[high] F3.1 — Offline `contacts remove` now unilaterally deletes trust state without ever sending `connection/revoke`**
`packages/cli/src/commands/contacts-remove.ts:36-54`

The new fallback path explicitly skips transport and removes the contact locally when tapd is unavailable. Before this change the CLI attempted `revokeConnection()` and only deleted after the revoke had been journaled; the repo guidance also treats revoke-before-delete as the recovery invariant for keeping both sides converged. With this code, a daemon outage during cleanup leaves the remote side `active` while the local side has already discarded the contact record, so the trust graph silently diverges and recovery becomes manual.

**Recommendation:** Fail closed when tapd is offline, or recreate the old short-lived runtime path so `connection/revoke` is still sent/journaled before local deletion.

**[high] F3.2 — `tap daemon start|stop|restart` can report success against the wrong process**
`packages/cli/src/lib/tapd-spawn.ts:53-136`

The lifecycle helper never proves that the spawned child actually bound the port or that the stopped child actually exited. `spawnTapdDetached()` accepts any pre-existing `.tapd.port` file as success, so a stale file or still-running old daemon can make `start`/`restart` look healthy even if the new child immediately dies. `stopTapdDetached()` then removes the pidfile after the timeout loop without checking whether the port file is still present or the process is still alive. The result is a split-brain control plane: the CLI can claim tapd restarted/stopped while another process still owns the same data dir.

**Recommendation:** Unlink stale port state before spawn, wait for a health check from the spawned PID instead of just any port file, and make stop fail if the process or port is still alive after the timeout.

**[high] F3.3 — tapd no longer reproduces the CLI's chain-aware signing/config behavior, so non-default-chain transfers regress**
`packages/tapd/src/bin.ts:22-82`

The daemon now loads config directly from `loadTrustedAgentConfigFromDataDir()` and constructs one `OwsSigningProvider` bound to `trustedAgentsConfig.chain` for the entire process. The old CLI transfer path built the provider per requested chain and loaded extra chain definitions (notably Taiko). After this migration, a Base-configured daemon handling `tap transfer --chain taiko` has neither the CLI's extra chain config nor a Taiko-scoped signer, so the command can fail as unsupported or hit the wrong OWS policy path even though the CLI still advertises Taiko support. The same startup-only load also drops CLI-side runtime overrides such as RPC/execution settings for transport-owned flows.

**Recommendation:** Load daemon config through the same chain-aware path the CLI uses, and create the signing/execution context per request chain instead of reusing a single startup-time provider bound to the default chain.

### Next steps
- Restore revoke-before-delete semantics for `tap contacts remove` or make the command refuse to mutate local state when tapd is unavailable.
- Harden daemon lifecycle management so start/stop/restart verify the actual child process and fail on stale sentinel files or incomplete shutdown.
- Rework tapd startup/execution wiring to honor the CLI's full chain/runtime config surface, especially Taiko and per-command transfer settings.

## Phase 4 — Hermes migration

**Target:** `80b6d2c..639df9c`
**Verdict:** needs-attention

No-ship: the Hermes migration no longer respects configured identities, no longer delivers notification context, and ships action payloads that tapd itself rejects.

### Findings

**[critical] F4.1 — Hermes actions ignore the configured identity/dataDir and always target the default tapd socket**
`packages/cli/assets/hermes/plugin/client.py:16-251`

`tap hermes configure` still writes named identities, and the Hermes tool schema still accepts an `identity` argument, but this client never reads Hermes `config.json` and never uses `identity` to select a target. It resolves a single socket from `TAP_DATA_DIR`/`~/.trustedagents`, then sends every action to that daemon. In any non-default or multi-identity Hermes setup, requests are silently executed against whichever agent owns that socket, so sends/transfers/pending approvals can hit the wrong identity.

**Recommendation:** Resolve the target identity from Hermes plugin config on each request, require `identity` when multiple entries exist, and derive the socket/dataDir from that resolved entry instead of a global default.

**[high] F4.2 — `/api/notifications/drain` is wired to a queue that nothing ever populates**
`packages/tapd/src/daemon.ts:64-295`

The daemon now exposes Hermes notifications by draining `this.notifications`, but the daemon never subscribes runtime events into that queue. `TapdRuntime` only publishes translated events to `EventBus`, and there is no corresponding `notifications.enqueue(...)` path in `packages/tapd/src`. The practical result is that Hermes `pre_llm_call` will see an always-empty notification list, so inbound messages, escalations, and pending approvals stop surfacing in operator context.

**Recommendation:** Add a real producer path from runtime events into `NotificationQueue` (including classification/deduping rules), then cover `/api/notifications/drain` with an integration test that observes a live TAP event end to end.

**[high] F4.3 — Several Hermes action payloads no longer match tapd's route contracts**
`packages/cli/assets/hermes/plugin/client.py:165-208`

The new Python client is not building the bodies that tapd expects. `request_meeting` only sends `{ peer, proposal: { title, duration, preferred, location } }`, but `/api/meetings` validates a full `SchedulingProposal` with `type`, `schedulingId`, and `slots`, so Hermes meeting requests now fail with 400s. The same mapping also omits `chain`/`toAddress` unless the caller supplies them for `request_funds` and `transfer`, even though the new routes require those fields and the old Hermes flow supplied defaults. This breaks previously supported Hermes actions on normal operator input.

**Recommendation:** Make the Python client reuse the same request-shaping/defaulting logic as tapd or the CLI: generate complete scheduling proposals, preserve previous defaults for transfer/funds fields, and add route-compatibility tests for every Hermes action against the real tapd handlers.

### Next steps
- Bind Hermes runtime selection to the configured identity entries instead of `TAP_DATA_DIR` alone.
- Restore a real notification pipeline into `/api/notifications/drain` before relying on Hermes pre-prompt context.
- Add end-to-end compatibility tests for each Hermes action against tapd's actual HTTP routes, not a fake socket stub.

## Phase 5 — OpenClaw refactor

**Target:** `639df9c..f775449`
**Verdict:** needs-attention

No-ship: this refactor cuts OpenClaw over to tapd surfaces that are not actually wired yet, and it also regresses core tool contracts (`request_meeting`, `autoGenerated`) with a real risk of targeting the wrong agent socket under isolated deployments.

### Findings

**[high] F5.1 — Socket discovery ignores `TAP_DATA_DIR`, so the plugin can bind to the wrong agent**
`packages/openclaw-plugin/src/tapd-client.ts:54-57`

`resolveSocketPath()` falls back straight to `$HOME/.trustedagents/.tapd.sock` when the plugin config omits `dataDir`/`socketPath`. The rest of this repo treats `TAP_DATA_DIR` as the default isolation override, and the Hermes client in this same branch honors it. In an OpenClaw deployment that starts tapd under `TAP_DATA_DIR`, this plugin will miss that socket and either fail or talk to the user's default daemon instead. Because `tap_gateway` can send messages, resolve approvals, and initiate transfers, that is a cross-agent isolation bug, not just a startup inconvenience.

**Recommendation:** Use the same resolution order as the rest of TAP: explicit `socketPath` > explicit `dataDir` > `process.env.TAP_DATA_DIR` > `~/.trustedagents`, and add a regression test for the env override.

**[high] F5.2 — The new tapd-backed notification/escalation path is unwired end-to-end**
`packages/openclaw-plugin/src/plugin.ts:23-67`

This branch removes the in-plugin queue and relies on `drainNotifications()` plus the SSE watcher. But I could not find any `NotificationQueue.enqueue()` call sites under `packages/tapd/src`, so `GET /api/notifications/drain` will stay empty. Separately, the watcher only wakes on `action.pending` and `connection.requested`, while `packages/tapd/src/runtime.ts` currently translates TAP events into `message.received`, `message.sent`, `action.requested`, `action.completed`, and `connection.established` only. In production that means OpenClaw neither wakes on real transfer/scheduling escalations nor injects `[TAP Notifications]` for them, so pending approvals become invisible unless an operator manually polls `list_pending`.

**Recommendation:** Do not ship the cutover until tapd actually enqueues notifications and emits the escalation event types the watcher expects, or keep the local queue/heartbeat path for now. Cover this with an end-to-end test against a real in-process tapd instead of the fake SSE/drain stubs.

**Note:** This is the same root cause as F4.2 — `/api/notifications/drain` is unwired. F4.2 and F5.2 should be treated as one fix with two downstream consumers.

**[high] F5.3 — `request_meeting` now sends a body that tapd rejects**
`packages/openclaw-plugin/src/tool.ts:179-187`

The old OpenClaw path built a full `SchedulingProposal` before dispatch. After the refactor, `request_meeting` forwards a flat `{ peer, title, duration, preferred, location, note }` object. The tapd route expects `{ peer, proposal: SchedulingProposal }` with fields like `type`, `schedulingId`, and `slots`, so `tap_gateway request_meeting` now fails instead of creating a meeting request.

**Recommendation:** Restore the proposal-building logic on the OpenClaw side, or make `OpenClawTapdClient.requestMeeting()` translate the flat tool params into the exact route contract and verify it with a real client-to-route test.

**Note:** Same root cause as F4.3 — `request_meeting` contract mismatch. The OpenClaw client and the Hermes Python client both got the body shape wrong in the same way.

**[medium] F5.4 — `autoGenerated` reply-loop protection is silently dropped on `send_message`**
`packages/openclaw-plugin/src/tool.ts:144-150`

The tool still advertises `autoGenerated` as the flag that prevents reply loops, and the client forwards it. But tapd's `/api/messages` route only accepts `{ peer, text, scope? }` and calls `service.sendMessage()` without the metadata/options object. ## v2 — SQLite conversation store

**Target:** `f775449..71f441a`
**Verdict:** needs-attention

No-ship: the SQLite migration can silently strand legacy conversations and it rewrites migrated metadata in ways that will misorder or mislabel existing threads.

### Findings

**[high] Fv2.1 — Migration marks itself complete even when some files failed to import**
`packages/core/src/conversation/sqlite-migration.ts:56-131`

Any malformed legacy JSON, transient read error, or DB insert failure leaves that conversation absent from SQLite forever. This path records per-file errors, then still archives the whole `conversations/` directory and unconditionally writes `conversation_logs_migrated_at`. Because callers only react to thrown exceptions, these partial failures are effectively silent. The likely result is user-visible conversation history disappearing after upgrade, with no automatic retry path.

**Recommendation:** Fail the migration closed: do not rename the source directory or set the completion flag unless every file imported cleanly. Surface `MigrationReport.errors` to callers and require an explicit retry/recovery path for partial migrations.

**[medium] Fv2.2 — Non-empty legacy conversations are re-timestamped and forced back to `active` during import**
`packages/core/src/conversation/sqlite-migration.ts:65-106`

For non-empty logs, the importer rebuilds the conversation by replaying messages through `logMessage()`. That creates the row from the first replayed message, overwrites `last_message_at` on each append, and never restores the source log's `startedAt`, `lastMessageAt`, or `status`. That is unsafe because legacy JSON is not guaranteed to be chronological; the old file-backed logger normalized on read rather than on disk. A migrated thread can therefore be permanently re-ordered in `/api/conversations`/CLI output, show the wrong transcript date, and lose any non-`active` status.

**Recommendation:** Sort imported messages by timestamp before replay, then explicitly update the conversation row from the source log's canonical metadata (`startedAt`, `lastMessageAt`, `lastReadAt`, `status`) after import. Add a regression test for an out-of-order legacy JSON file.

### Next steps
- Make the migration atomic or fail-closed, and surface partial-import errors instead of silently proceeding.
- Preserve canonical conversation metadata during import and add regression coverage for out-of-order legacy histories.
