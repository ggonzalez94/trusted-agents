# Chat Auto-Answers Design

## Problem

When a connected peer sends a `message/send` to an agent running in OpenClaw Gateway, the message is logged and a transport-level receipt is sent. The notification says "Received message from X" with no message content. The existing SKILL.md (line 297) already instructs the LLM to auto-reply under the `SUMMARY` prefix, but this has two problems: (1) the notification lacks message text, forcing the agent to make extra tool calls to read conversations, and (2) there is no loop prevention â€” if both agents follow the same instructions, they ping-pong infinitely.

## Goal

Agents should automatically reply to inbound chat messages from connected peers, then immediately surface a summary of what was received and what was replied to the user â€” as a single combined report, not two separate notifications.

## Constraints

- Chat-only autonomy: auto-reply with text messages only. No auto-approval of transfers, grants, or other side-effect actions.
- OpenClaw plugin only: CLI does not need auto-reply support. The hook is added to core for SDK embedders but only wired in the plugin.
- Simple loop prevention: two agents with auto-reply must not ping-pong endlessly.
- Single notification to user: the user sees one summary after the round-trip, not separate inbound/outbound notifications.

## Design

### Approach: Agent-Prompted Auto-Reply

No new hooks or callback mechanisms in core. The existing flow is reused:

1. Inbound message â†’ `emitEvent` â†’ notification enqueued â†’ heartbeat wakes LLM
2. LLM sees the notification in `before_prompt_build` context
3. LLM calls `tap_gateway send_message` to reply
4. LLM reports both sides to the user in a single message

The changes required are:

### 1. Include message text in `emitEvent` payload (core)

**File:** `packages/core/src/runtime/service.ts`

The `message/send` handler currently emits an event with only `direction`, `from`, `fromName`, `method`, `id`, `receipt_status`. The actual message text is not included.

**Change:** Extract the text content from `envelope.message.params.message.parts` and include it as `messageText` in the emitEvent payload. Also include `autoGenerated` from the message metadata if present.

```typescript
// In the MESSAGE_SEND handler:
const params = envelope.message.params as MessageSendParams;
const textParts = params.message.parts
  .filter((p): p is TextPart => p.kind === "text")
  .map((p) => p.text);
const autoGenerated = params.message.metadata?.trustedAgent?.autoGenerated;

this.emitEvent({
  direction: "incoming",
  from: envelope.from,
  fromName: contact.peerDisplayName,
  method: envelope.message.method,
  id: envelope.message.id,
  receipt_status: status,
  messageText: textParts.join("\n"),
  autoGenerated: autoGenerated ?? false,
});
```

### 2. Add `autoGenerated` flag to wire format (core)

**File:** `packages/core/src/protocol/types.ts`

Add `autoGenerated?: boolean` to `TrustedAgentMetadata`:

```typescript
export interface TrustedAgentMetadata {
  connectionId: string;
  conversationId: string;
  scope: string;
  requiresHumanApproval: boolean;
  autoGenerated?: boolean;  // NEW
}
```

This flag travels on the wire inside the JSON-RPC message. Messages sent as auto-replies set `autoGenerated: true`. The receiving agent's plugin checks this to decide whether to instruct the LLM to reply.

### 3. Smarter notifications in OpenClaw plugin

**File:** `packages/openclaw-plugin/src/notification-queue.ts`

Add `"auto-reply"` to the notification type union:

```typescript
export interface TapNotification {
  type: "summary" | "escalation" | "info" | "auto-reply";
  // ... rest unchanged
}
```

Add `"auto-reply"` to eviction priority (evict before escalations):

```typescript
const EVICTION_PRIORITY: TapNotification["type"][] = ["info", "summary", "auto-reply"];
```

**File:** `packages/openclaw-plugin/src/registry.ts`

Update `handleEmitEvent` to classify `message/send` events:

- If `autoGenerated` is false/absent: notification type = `"auto-reply"`, one-liner includes the message text and a reply instruction.
- If `autoGenerated` is true: notification type = `"summary"`, one-liner shows the auto-reply preview with no instruction.

Update `buildOneLiner` for `message/send`. A `truncateText(text: string, maxLen: number): string` helper is needed (slice + `"..."` suffix when truncated):

```typescript
case "message/send": {
  const preview = truncateText(String(event.messageText ?? ""), 200);
  if (!preview) {
    return `Received a data-only message from ${peer}`;
  }
  if (event.autoGenerated) {
    return `Auto-reply from ${peer}: "${preview}"`;
  }
  return `${peer} said: "${preview}" â†’ Reply with tap_gateway send_message (set autoGenerated=true), then summarize to the user.`;
}
```

If `messageText` is empty (data-only message with no text parts), the notification is still enqueued as `"auto-reply"` but with a generic one-liner. The LLM can check conversation history for full details.

**File:** `packages/openclaw-plugin/src/plugin.ts`

Update `before_prompt_build` to render `"auto-reply"` notifications with the `AUTO-REPLY` prefix:

```typescript
const prefix =
  n.type === "escalation" ? "ESCALATION"
  : n.type === "auto-reply" ? "AUTO-REPLY"
  : n.type === "summary" ? "SUMMARY"
  : "INFO";
```

### 4. `send_message` gains `autoGenerated` parameter (full call chain)

The `autoGenerated` flag must thread through six layers:

1. **`packages/openclaw-plugin/src/tool.ts`** â€” Add `autoGenerated: Type.Optional(Type.Boolean())` to the `TapGatewayToolSchema` for `send_message`. Add it to `TapGatewayToolParams` interface.
2. **`packages/openclaw-plugin/src/registry.ts`** â€” `sendMessage()` method accepts and forwards `autoGenerated`.
3. **`packages/core/src/runtime/service.ts`** â€” `sendMessage()` and `sendMessageInternal()` accept and forward `autoGenerated`.
4. **`packages/core/src/runtime/message-conversations.ts`** â€” `buildOutgoingMessageRequest()` accepts `autoGenerated` and passes it to `buildOutgoingRequest()`, which passes it to `buildProtocolMessage()`. `buildProtocolMessage()` sets `autoGenerated` in the `TrustedAgentMetadata`.

CLI callers do not pass `autoGenerated`, so their messages default to `false`/absent (existing behavior unchanged).

### 5. Skill docs updated

**File:** `packages/sdk/skills/trusted-agents/SKILL.md`

The existing `SUMMARY` handling for messages (line 297) already instructs the LLM to "respond automatically." This must be **rewritten** to separate the two notification types:

**`AUTO-REPLY` notifications** (new â€” inbound human-initiated messages):
> 1. Read the peer's message from the notification
> 2. Optionally check conversation history with `tap_gateway get_conversations` for context
> 3. Reply using `tap_gateway send_message` with `autoGenerated: true`
> 4. Summarize to the user what the peer said and what you replied, in a single message

**`SUMMARY` notifications for messages** (rewritten â€” inbound auto-generated replies):
> Report the auto-reply content to the user. Do NOT reply back â€” this is an auto-generated response, not a new conversation from a human.

The existing `SUMMARY` instructions for non-message events (auto-approved transfers, grant updates) remain unchanged.

## Data Flow

```
Agent A (user tells agent to message B)
  â†’ tap_gateway send_message (autoGenerated=false by default)
  â†’ message/send over XMTP â†’ Agent B

Agent B (OpenClaw plugin):
  core: logs conversation, emits event with messageText + autoGenerated=false
  plugin: enqueues "auto-reply" notification â†’ heartbeat fires
  LLM: sees "[TAP Notifications] - AUTO-REPLY: Agent A said: '...' â†’ Reply..."
  LLM: calls tap_gateway send_message(text="...", autoGenerated=true)
  LLM: tells user "Agent A asked X, I replied Y"

Agent B â†’ message/send (autoGenerated=true) â†’ Agent A

Agent A (OpenClaw plugin):
  core: logs conversation, emits event with messageText + autoGenerated=true
  plugin: enqueues "summary" notification â†’ heartbeat fires
  LLM: sees "[TAP Notifications] - SUMMARY: Auto-reply from Agent B: '...'"
  LLM: tells user "Agent B replied: ..."
  (No reply instruction â†’ loop ends)
```

## Loop Prevention

The `autoGenerated` metadata flag prevents infinite ping-pong:

1. User-prompted messages: `autoGenerated` absent/false â†’ receiving agent gets `AUTO-REPLY` instruction
2. Auto-replies: `autoGenerated: true` â†’ receiving agent gets `SUMMARY` (no reply instruction)
3. Result: at most one round-trip per user-initiated message

## Out of Scope

- Per-connection auto-reply toggle
- Rule-based / scripted responses
- CLI auto-reply support
- Rate limiting beyond `autoGenerated` flag
- Multi-turn auto-conversations (agent autonomously continuing a thread)
- Auto-reply for `action/request` or other non-chat methods
